## 1. FunASR 调研结论（设计依据）

### 1.1 官方模型矩阵（来自 funasr.com 官方页）

| 模型 | 参数量 | 语种 | 设备 | 实时率 | 备注 |
|---|---|---|---|---|---|
| **SenseVoice Small** | 234M | 50+（含中文） | **CPU 友好** | 170× | 非自回归；含情感/音频事件检测 |
| **Paraformer-zh Large** | 220M | 中文 + 粤语 | CPU | 120× | 中文精度最高；内置标点 + 时间戳 |
| **Fun-ASR-Nano** | 800M | 31 语种 | **需 GPU/vLLM** | 340× | LLM-ASR，桌面 CPU 不可行 |

### 1.2 河南话（中原官话）现状 —— **关键缺口**

- **三者均不原生支持河南话**。可落地补法：
  1. **社区微调模型**：Modelscope/魔搭上有河南话 Paraformer 微调权重，检索后接入；
  2. **普通话兜底 + 纠错补词**：用 Paraformer-zh / SenseVoice（中文）识别，再用现有 `correct_text` + 方言词库做后处理，把"中字"校正为"方言词"（如 中原官话与普通话高度近似，此法可用）；
  3. **自训轻量模型**：若社区无满意权重，用少量河南话录音微调 Paraformer（需数据，列为 M1 后续）。

### 1.3 打包/体积 —— 核心矛盾与解法

- FunASR 完整版依赖 `torch` + `modelscope`，体量约 **1GB+**，直接塞进当前 onedir 包会让体积从 ~700MB 飙到 ~2GB，且 torch 在 PyInstaller 下易踩坑。
- **官方推荐边缘部署 = ONNX + onnxruntime（免 torch）**：FunASR 支持把 Paraformer/SenseVoice 导出为 ONNX，用 `onnxruntime` 后端推理，无需 torch。模型 int8 量化后约 **几十~两百 MB**，运行时 + onnxruntime 约 50MB。
- **结论**：v2.0 主引擎走 **FunASR 模型（ONNX 导出）+ onnxruntime（torch-free）**；把完整 torch 版 FunASR 作为「高精度可选插件」按需外挂下载，不进主包。

## 2. 总体架构（目标态）

```
┌──────────────────────────────────────────────────────────┐
│  表现层 (PySide6)  main / page_main / gallery / records /  │
│  map / game / settings  ·  主题(暗/亮) · 托盘驻留          │
├──────────────────────────────────────────────────────────┤
│  业务层 (backend.py / worker.py)  解析 · 纠错 · TTS · 编排  │
├──────────────────────────────────────────────────────────┤
│  引擎层 (asr/)  可插拔 ASR：FunASR(离线) / Xunfei(云端) /   │
│  Demo(词库)   ·  TTS：Edge(在线) / 离线TTS(可选)           │
├──────────────────────────────────────────────────────────┤
│  数据层 (SQLite)  历史 · 收藏 · 词库包 · 学习进度 · 设置    │
├──────────────────────────────────────────────────────────┤
│  资源层  dialect_dict*.json(多词库) · asr_models/(onnx) ·   │
│  tts_cache/ · recorder(16k mono wav)                      │
└──────────────────────────────────────────────────────────┘
```

- **识别引擎可插拔**：统一接口 `ASREngine.transcribe(wav)->text`，设置里切换三种后端，默认离线 FunASR。
- **数据流**：`recorder.py` 录 16k mono wav → `worker.py` 后台线程调 `ASREngine` → `backend.correct_text` 后处理 → 页面展示 + 写 SQLite。

- **目标**：双击即用、纯离线、免密钥的方言/中文语音识别。
- **技术方案**：
  1. 新增 `asr/__init__.py` + `asr/base.py`（`ASREngine` 抽象：`load()` / `transcribe(wav_path)->str` / `stream_start()` / `stream_feed()`）。

## 4. 识别引擎细化设计（M1 核心）

```python
# asr/base.py（示意）
class ASREngine:
    name: str = "base"
    def load(self): ...                       # 懒加载模型到内存
    def transcribe(self, wav_path: str) -> str: ...
    def stream_start(self): ...
    def stream_feed(self, chunk: bytes) -> str: ...   # 返回增量文本
    def unload(self): ...

# backend.py 选择
def get_asr_engine(kind: str = "offline") -> ASREngine:
    if kind == "offline": return FunASREngine()   # 默认
    if kind == "cloud":   return XunfeiEngine()
    return DemoEngine()
```

- **默认离线引擎实现要点**：
  - `FunASREngine.load()`：从 `assets/asr_models/paraformer_zh.onnx` + `fsmn_vad.onnx` 构建 onnxruntime session（CPU EP）。**首次调用才加载**（沿用 v1 懒加载思路，不拖慢启动）。
  - 录音 → `recorder.py` 产 16k mono 16-bit wav → 送入 session → 文本。
  - 后处理：调用现有 `backend.correct_text` 做方言词校正。
- **与现有代码衔接**：现有 `backend.transcribe_audio(...)` 是 Xunfei 入口；改为内部路由到 `get_asr_engine(setting.backend).transcribe(...)`，对外接口不变，页面无感切换。

## 9. M1 Spike 实测结论（2026-07-08 ✅ 通过）

在 managed venv（`asr_spike`，Windows 风格 venv：`Scripts/python.exe`）完成 FunASR Paraformer 离线识别验证。

### 9.1 关键事实修正：模型格式
- 下载的 `sherpa-onnx-paraformer-zh-2023-09-14.tar.bz2`（234MB，字节数 234051698）解压后**只有一个 `model.int8.onnx`（243MB）+ tokens.txt + am.mvn**，**没有** sherpa-onnx 的 encoder/decoder/joiner 分体文件。
- 这是 **FunASR 官方导出的 Paraformer ONNX**（输入 `speech[B,T,560]`，输出 `logits/token_num/us_alphas/us_cif_peak`），不是 sherpa 的包装格式。因此**不能用 `sherpa_onnx.OfflineRecognizer.from_paraformer`**，必须走**原生 onnxruntime 推理**（引擎 `asr/funasr_onnx_engine.py` 已按官方 `test-paraformer-onnx.py` 重写）。
- torch-free 已验证：运行时仅依赖 `onnxruntime` + `kaldi_native_fbank` + `soundfile` + `scipy` + `numpy`，**无需 torch / funasr 完整版**。

### 9.2 特征与解码管线（对齐官方参考）
1. 读 wav（soundfile，16k mono；非 16k 用 scipy 重采样）。
2. kaldi_native_fbank 提 80-bin fbank（dither=0, snip_edges=False）。
3. LFR 窗：window=7, shift=6 → 特征维 560。
4. CMVN：从 `am.mvn` 读两段 `<LearnRateCoef>` 向量，第一段加、第二段乘。
5. onnxruntime 推理（CPU EP，intra/inter threads=1）。
6. 解码：`logits.argmax(-1)[:token_num]`，跳过 token 0/2（blank/unk），映射 `tokens.txt`。

### 9.4 已知短板（产品化需处理）
- 英文/BPE 子词会带 `@@` 连接符（如 `ye@@ster@@day`），朴素 argmax 未做子词合并；中英混合场景需补 BPE 拼接逻辑。
- 朴素 argmax 未走完整 CIF + 自注意力解码，个别字可能略偏；纯中文场景已足够。
- 当前为 int8 单模型；如需更高精度可换 fp32 `model.onnx`（本包未含，需另下）或社区河南话微调权重。

### 9.5 结论
M1 spike **通过**：FunASR Paraformer-ONNX + onnxruntime 可在本机纯离线、torch-free 跑通中文/方言识别，速度远快于实时。

### 9.8 BPE 子词合并（2026-07-08 ✅ 完成）——修复 §9.4 已知短板
`asr/funasr_onnx_engine.py` 的 `_decode` 重写：
1. 跳过 `<blank>`/`<s>`/`<s>`/`<unk>` 等特殊 token（原代码漏跳 `<s>`，argmax 命中会输出字面 `<s>`）；
2. 合并 `@@` 续接子词（`ye@@`+`ster@@`+`day` → `yesterday`）；
3. 英文词（全 ASCII）之间补空格，汉字/CJK 之间不补（该模型 vocab 无 `▁` 空格标记）。
- **验证**：单元合成 token 序列 5 例全对（合并/跳特殊/英文空格）；真实模型对中文语音识别与改前**逐字一致**（无 `@@`/无 `<s>`/无异常空格）；对 edge_tts 合成英文语音 `hello world this is a test of speech recognition` 正确识别且词间空格正确（旧代码会输出 `@@` 乱码）。`py_compile` 全绿。
- **说明**：本模型 BPE vocab 无 `▁`，英文词内无 `@@` 续接时按一词合并（如 `hello`+`world` 无 `@@` 连写），仅当两个独立非 `@@` 英文词相邻才补空格；属该模型 tokenizer 限制，已是最优处理。
